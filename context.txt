Project: Ergonomic Comic Translator

Purpose:
- Bun-driven OCR pipeline for comic page images and PDF files.
- Writes normalized OCR JSON to a configurable output path.
- Uses Poetry-managed Python for PaddleOCR runtime work.
- Post-OCR text removal via local Python pipeline (manga-text-cleaner + lama inpainting).
- Post-OCR translation via local Ollama LLM (page-by-page with full context).

Current architecture:
- System layer: WSL2/Linux, Homebrew, visible NVIDIA and/or AMD GPUs, CUDA driver passthrough from Windows, optional ZLUDA for AMD GPU acceleration.
- Python layer: Poetry + pyenv-managed Python 3.12, two domain packages under src/python/ invoked as modules (python -m):
  - ocr/ package: PaddleOCR runtime (runner.py), PDF page rendering (pdf.py), polygon geometry helpers (geometry.py)
  - textless/ package: text removal pipeline (runner.py), inpainting strategy dispatch (inpainting/), shared utilities (utils/)
- Bun layer: modular OCR code under src/ocr/, modular API code under src/api/, input discovery, runtime config, PDF preparation, batch OCR execution with multi-GPU support, logging, upload intake, and final JSON merge.
- Text cleaner layer: separate Python venv (.tmp/text-cleaner-venv/) using the textless/ package with manga-text-cleaner and lama-cleaner for text removal using OCR bounding boxes.
- Translation layer: Bun-driven Ollama integration (src/api/services/translate-processing.ts) translating OCR pages page-by-page using a local LLM. Each page call includes full OCR and accumulated translations as context. Output written incrementally to .tmp/translated/{scope}/translated.json.

Important current behavior:
- Shared path config lives in src/config.ts.
- Bun accepts image files and PDF files from the configured OCR file root; the default is .tmp/ocr/.
- PDF inputs are first rendered into per-page PNG files via src/python/pdf_splitter.py.
- All pages for a given scope are processed in a single batch Python process by default (one model load).
- When OCR_CONCURRENCY > 1 and multiple NVIDIA GPUs are available, pages are split round-robin across parallel GPU batches (one Python process per GPU).
- On a single GPU, OCR_CONCURRENCY has no effect — all pages run in one batch to avoid VRAM contention.
- Single PDF files are prepared page-by-page, then all page images are batched together.
- OCR output is written under .tmp/ocr_output/{scope} by default.
- OCR preparation artifacts and per-job JSON files are preserved under .tmp/ocr_output/{scope}/artifacts.
- Direct Bun OCR uses scope ocr; files discovered under the API upload root use their uploadId as the output scope.
- Bun API server accepts image, PDF, and ZIP uploads.
- Bun API server uses GET /api/files to list stored files and POST /api/upload to accept new uploads.
- Bun API server exposes POST /api/ocr/{uploadId}, GET /api/ocr, DELETE /api/ocr/{uploadId}, and GET /api/ocr/{uploadId} for OCR queue management.
- Bun API server exposes POST /api/textless/{uploadId}, POST /api/textless/{uploadId}/{page}, GET /api/textless, and GET /api/textless/{uploadId} for textless queue management.
- Bun API server exposes POST /api/translate/{uploadId}, POST /api/translate/{uploadId}/{page}, GET /api/translate, GET /api/translate/{uploadId}, GET /api/translate/{uploadId}/{page}, and PUT /api/translate/{uploadId}/{page} for translation queue management and per-page read/write.
- OCR queue state is persisted in .tmp/program/ocrQueue.json by default.
- Textless queue state is persisted in .tmp/program/textlessQueue.json by default.
- Translation queue state is persisted in .tmp/program/translateQueue.json by default.
- OCR queue states are Ready, Queued, Processing, and Completed.
- Textless queue states are Ready, Queued, Processing, and Completed (shared JobStatus type).
- OCR and textless queues share a unified JobStatus type defined in src/api/interfaces/job-status.ts.
- If no OCR task is active, a queued upload starts immediately; otherwise it waits for the current upload to finish.
- Once an upload reaches Completed, later POST /api/ocr/{uploadId} requests do not re-add it to the queue.
- Textless queue requires OCR Completed status; uploads can be re-queued. Per-page tracking (pending/completed/failed). One uploadId at a time.
- Translation queue requires OCR Completed status; uploads can be re-queued. Per-page tracking (pending/completed/failed). One uploadId at a time. Pages translated sequentially with sliding window context (last TRANSLATE_CONTEXT_PAGES translated pages, default 8). If whole-page translation fails after 3 retries, automatically falls back to translating each line individually (2 retries per line).
- ZIP uploads are extracted into OCR-readable files under the configured API upload directory; the default is .tmp/upload/.
- Upload metadata is persisted through an API repository abstraction backed by .tmp/program/uploadRecord.json.
- By default, OCR staging and API uploads are separate: OCR scans .tmp/ocr while API stores uploads in .tmp/upload.
- OCR preparation copies images or splits PDFs into .tmp/ocr_prepare/{scope}/.
- OCR output is written to .tmp/ocr_output/{scope}/ocr_output.json.
- Text-removed images are written to .tmp/textless/{scope}/ with same filenames as source.
- Translation output is written to .tmp/translated/{scope}/translated.json.
- Per-page translation edits are read/written via GET/PUT /api/translate/{uploadId}/{page}; the UI saves inline edits back to the same translated.json.
- Queue state is stored in .tmp/program/ocrQueue.json.
- Textless queue state is stored in .tmp/program/textlessQueue.json.
- Final output is merged back in stable order:
- by original input file order
- then by original PDF page order
- then pageNumber is reassigned sequentially in the merged JSON

Folder structure under .tmp/:
- .tmp/program/: internal state (uploadRecord.json, ocrQueue.json, textlessQueue.json, translateQueue.json)
- .tmp/upload/{uploadId}/: raw uploaded files per upload batch
- .tmp/ocr_prepare/{scope}/images/: all prepared images for OCR (copied loose images, extracted ZIP images, split PDF pages)
- .tmp/ocr_output/{scope}/ocr_output.json: final merged OCR JSON
- .tmp/textless/{scope}/: text-removed images (same filenames as source)
- .tmp/translated/{scope}/translated.json: translated lines JSON
- .tmp/text-cleaner-venv/: separate Python venv for text removal pipeline

Data model:
- uploadId (UUID) is the sole identifier across the entire pipeline. Generated once per POST /api/upload request.
- UploadRecord: { uploadId, sourceType, originalName, storedName, storedPath, relativePath, contentType, size, createdAt, archiveName, archiveEntryName }
  - All files from one upload share the same uploadId. No per-file ID.
  - sourceType is image, pdf, zip, or zip-entry.
- OcrJobRecord: { uploadId, status, outputFile, createdAt, updatedAt, startedAt, completedAt, lastError }
  - One record per uploadId. Keyed and upserted by uploadId.
- TranslationJobRecord: { uploadId, status, outputFile, targetLanguage, pageRecords[], createdAt, updatedAt, startedAt, completedAt, lastError }
  - One record per uploadId. Keyed by uploadId.
  - pageRecords tracks per-page status: pending / completed / failed.
- TranslationOutput is an array of TranslatedPage: { pageNumber, lines: TranslatedLine[] }
- TranslatedLine: { lineIndex, translated }
- uploadId flows through: upload → OCR queue → ocr_prepare/{uploadId}/ → ocr_output/{uploadId}/ → textless/{uploadId}/ → translated/{uploadId}/

Frontend component architecture (src/frontend/):
- All pages and components use Container / View split + React Context for cross-cutting state.
- OcrPreviewPanel uses OcrEditorContext (showTranslation, showBoxes, polygonBgColor, onPolygonBgColorChange, lines, translatedLines, selectedLineIndex, etc.) shared between Container, sub-components, and hooks.
- EditorSnapshot and useEditorHistory track OCR lines + translatedLines together for unified undo/redo.
- SvgOverlay renders polygon drag handles + optional translation overlay (semi-transparent bg + fitted text inside each polygon).
- polygonTextLayout.ts (src/frontend/components/OcrPreviewPanel/utils/) computes AABB, binary-search max font size, horizontal wrap, and vertical right-to-left column layout; CJK chars ≈1em wide, ASCII ≈0.6em wide.
- ImageToolbar has 3 toggle groups: image mode (original/textless), show/hide boxes, show/hide translation (default hidden); also has ARGB polygon background color picker (MUI Popover with native <input type="color"> for RGB + MUI Slider for alpha + live preview swatch).
- polygonBgColor (ARGB rgba string) is stored in OcrPreviewPanelContainer state; persisted to localStorage keyed by `polygonBgColor:${uploadId}`; exposed via OcrEditorContext (polygonBgColor, onPolygonBgColorChange) and included in OcrPreviewPanelRef.getViewState().
- SvgOverlay uses polygonBgColor from context for polygon fill and polygonTextColor(polygonBgColor) from src/frontend/config.ts to automatically choose white or #1a1a1a text color based on WCAG relative luminance (alpha-composited on white).
- src/frontend/config.ts exports DEFAULT_POLYGON_BG_COLOR constant and polygonTextColor(bgColor: string) helper.
- TranslationEditor (bottom-right panel) shows editable translated text for the selected OCR line.
- TranslateDialog (Container + View) lets user pick model, language, and scope (all pages / current page); polls translation job status until complete.
- UploadDialogContainer exposes uploadError state; UploadDialogView renders an Alert on failure.

Current OCR pipeline details:
- src/python/paddleocr_runner.py supports both PaddleOCR and PaddleOCR-VL-1.5.
- Standard OCR path:
- uses PaddleOCR(..., use_textline_orientation=True)
- returns line confidence directly from rec_scores
- VL path:
- uses PaddleOCRVL(pipeline_version="v1.5", ...)
- normalizes parsing_res_list / spotting_res into the shared output schema
- adds orientation for each line/block
- VL does not expose confidence values

Output schema expectations:
- Top level fields:
- source
- ocrEngine
- ocrModel
- language
- device
- generatedAt
- pageCount
- pages[]
- Each page contains:
- pageNumber
- fileName
- filePath
- lines[]
- Each line contains:
- lineIndex (0-based index within the page)
- text
- box as [x1, y1, x2, y2]
- orientation
- polygon (array of [x, y] coordinate pairs)

- Python-Bun message channel protocol:
  - stderr = logging channel: Python scripts write lines with [INFO]/[WARN]/[ERROR] prefixes to sys.stderr
  - stdout = data channel: used for structured result data (or unused when writing to file)
  - Poetry scripts (OCR, PDF splitter) use Bun.spawn() with piped stdout+stderr, invoked as `poetry run python -m ocr.runner` / `poetry run python -m ocr.pdf`
    - stderr is streamed line-by-line via readStreamLines() and forwarded through forwardPythonLine()
    - stdout is drained fully as string; returned to caller
    - both streams consumed concurrently with Promise.all to prevent pipe-buffer deadlocks
  - Text cleaner uses Bun $ shell tag with .nothrow(), invoked as `python -m textless.runner`; stderr forwarded via forwardPythonLogs() after exit
  - ocr.runner: writes JSON to --output file (not stdout); stderr has progress lines
  - ocr.pdf: writes JSON manifest to --output file (not stdout)
  - textless.runner: last non-empty stdout line is JSON {success, outputPath} or {success, error}
  - OCR progress extraction: regex /\[INFO\] Processing page (\d+)\/(\d+)/ on stderr onLine callback triggers onProgress(current, total)
  - Non-zero exit codes throw Error("{prefix} failed with exit code {code}")
  - PADDLE_PDX_DISABLE_MODEL_SOURCE_CHECK=True set inside ocr/runner.py
  - LD_LIBRARY_PATH extended with ZLUDA path for AMD GPU support via resolveZludaEnv()
  - PYTHONPATH set to src/python/ for all Python invocations so relative imports resolve correctly

- Uses a local Python pipeline with manga-text-cleaner for mask generation and lama_large for inpainting.

Known caveats / editing guidance:
- Keep Bun responsible for filesystem, orchestration, logging, and config.
- Keep Bun API code inside src/api/ and keep OCR orchestration inside src/ocr/.
- Keep Python code organized as packages under src/python/ (ocr/ and textless/). Use relative imports within packages.
- Keep Bun invoking Python via `python -m <module>` with PYTHONPATH, never as standalone script files.
- Keep Python responsible for OCR runtime work and PDF rendering.
- Keep Bun OCR code inside src/ocr/ rather than moving orchestration back into src/main.ts.
- For upload persistence changes, prefer extending the repository abstraction rather than coupling server routes directly to file storage.
- When adding Bun OCR types, prefer creating a dedicated file under src/ocr/interfaces/ and exporting it through src/ocr/interfaces/index.ts.
- When adding API types, prefer creating a dedicated file under src/api/interfaces/ and exporting it through src/api/interfaces/index.ts.
- When adding API modules, prefer exporting them via src/api/index.ts or folder-level index.ts files.
- When adding Bun OCR submodules, prefer exporting them via src/ocr/index.ts or the relevant folder index.ts.
- Preserve the normalized output schema unless a task explicitly requires changing it.
- Preserve stable merge ordering for concurrent OCR results.
- Prefer small targeted edits; avoid undoing the WSL2 + Poetry + Bun layered design.
- Do not reintroduce Docker unless explicitly requested.
- Ollama runs externally (e.g. on a Windows host or separate server) — configure via OLLAMA_HOST env var.
- Translation uses streaming mode (stream:true) with per-page calls to avoid LLM output truncation.
- Each translation call includes the full OCR text and all already-translated pages in the system prompt for translation consistency.
- extractFirstObject() in translate-processing.ts is the robust JSON parser — do not replace with plain JSON.parse.

- All components follow the Container / View pattern:
  - *Container.tsx — all hooks, state, business logic. No raw JSX beyond wiring to the View.
  - *View.tsx — pure render. Receives props or reads from context. May use useTranslation/useTheme.
  - index.tsx — thin re-export only.
- Complex components use React Context to avoid prop drilling between Container, View, and sub-components.
- Complex components extract logic into custom hooks under hooks/ and sub-views under components/.

Coding standards:
- TypeScript strict mode with verbatimModuleSyntax. Use `import type` for type-only imports.
- Directory barrel imports omit /index.ts: `from "../interfaces"`. Direct file imports keep .ts extension: `from "../config.ts"`.
- One interface/type per file under interfaces/ directories, with barrel index.ts for re-export.
- Files use kebab-case. Interfaces/types use PascalCase. Functions/variables use camelCase.
- Shared status types use a single JobStatus union (Ready | Queued | Processing | Completed) across all queue services.
- Queue services follow the same state machine: Ready → Queued → Processing → Completed.
- Repository interfaces are async (seam for future database migration). Services receive repositories via constructor injection.
- Scripts in src/scripts/ are thin CLI wrappers — import logic from service modules, no duplicated business logic.
- Shared utilities extracted to dedicated files: src/scripts/shell-utils.ts, src/api/utils/time-utils.ts, etc.
- Environment variables resolved once in src/config.ts with sensible defaults.
- No barrel index.ts re-exports implementation details — only public API surfaces.
- Frontend components follow Container/View pattern. Complex components use React Context + custom hooks + sub-components.
- Keep frontend component logic in hooks/ and sub-views in components/. Do not put business logic in View components.
- OcrPreviewPanel sub-components read from focused contexts via useOcrLines(), useOcrView(), useOcrTranslation(), useOcrActions() — each component only subscribes to the contexts it needs. Do not use a monolithic context.
